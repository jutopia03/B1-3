# 프로젝트 1 — 자동화 도구 비교 구현 (Make vs n8n)

## 1. 개요

동일한 자동화 워크플로우를 **Make**(클라우드 SaaS)와 **n8n**(로컬 자가호스팅) 두 도구로 각각 구현하고, 구현 과정에서 실측한 차이를 근거로 비교 분석한 문서입니다.

| 항목 | 내용 |
|---|---|
| 워크플로우 | 우송대학교 공지사항 3개 게시판 감시 → 학사/공모전/기타 분류 → 텔레그램 알림 + Google Sheets 기록 |
| 비교 도구 | Make (무료 플랜) / n8n (로컬 자가호스팅) |
| 실행 주기 | 양쪽 모두 매일 09:00 1회 (통제변인) |
| 수집 대상 | www / cs / swhit 3개 게시판, 각 10건 = 30건 |

### Zapier를 제외한 이유

당초 Make·Zapier·n8n 3종 비교를 검토하였으나, **Zapier 무료 플랜은 2-step Zap만 허용**하므로 과제 요구사항(Trigger 1개 + Action 2개 이상 + 조건 분기 1개 이상)을 구조적으로 충족할 수 없었습니다. 멀티스텝·필터·웹훅이 모두 유료 기능입니다.

과제 제약이 "무료 플랜으로 완수 가능한 조합을 우선 고려한다"이므로, Zapier는 비교 대상에서 제외하고 **제약 분석 사례**로만 기록합니다.

## 2. 공통 요구사항 충족 현황

| 요구사항 | Make | n8n |
|---|---|---|
| Trigger 1개 이상 | Schedule (매일 09:00) | Schedule (매일 09:00) |
| Action 2개 이상 | Google Sheets Add a Row, Telegram Send Message, Google Sheets Update a Cell | Google Sheets Append, Telegram Send Message, Google Sheets Update Row |
| 조건 분기 1개 이상 | Router (4경로) | Switch (3경로) + Filter |
| **각 분기 경로 1회 이상 실행** | 학사 16 / 공모전 10 / 기타 4 / 상태갱신 3 | 학사 16 / 공모전 10 / 기타(Fallback) 4 |

첫 실행에서 30건 전부를 통과시켜 모든 분기 경로가 실제로 발동하는 것을 확인하였습니다. 이를 위해 `_state` 시트의 `last_seen_id` 초기값을 "각 소스의 현재 최소 ID − 1"로 설정하였습니다.

## 3. 대상 사이트 분석

세 사이트 모두 동일한 JSP 기반 CMS를 사용하며 URL 패턴이 같습니다.

```
목록: /board/index.jsp?code={보드코드}
상세: /board/read.jsp?id={게시글ID}&code={보드코드}
```

| source | 도메인 | 보드코드 | 성격 | 접근 |
|---|---|---|---|---|
| www | www.wsu.ac.kr | community0101 | 학사·행정 위주 | SSO 우회 필요 |
| cs | cs.wsu.ac.kr | cs0501 | 학사 행정 | 익명 직접 |
| swhit | swhit.wsu.ac.kr | swhit0501a | 공모전·프로그램 | 익명 직접 |

RSS/Atom 피드가 제공되지 않아 **스케줄 트리거 + HTTP 요청 + HTML 파싱** 방식을 채택하였습니다.

`read.jsp?id=NNNNNN`의 `id`를 고유 키로 사용합니다. 세 사이트가 ID 공간을 공유하며 전 사이트 통합으로 증가하지만, 게시판별 진행 속도가 달라 `last_seen_id`는 **소스별로 따로 관리**합니다. 이 설계 판단이 후술할 비교 항목 8의 배경이 됩니다.

## 4. 구현 결과

### 4.1 Make 시나리오

```
[Schedule 09:00]
  ↓
[HTTP: sso_identify]        Allow redirects: No / Share cookies: Yes / Parse response: No
  ↓
[Iterator: split("www;cs;swhit"; ";")]
  ↓
[Tools: Set multiple variables]   switch()로 소스별 board_url / read_base / code_param / cell 계산
  ↓
[HTTP: board]
  ↓
[Google Sheets: Get a Cell]       last_seen_id 조회
  ↓
[Text parser: Match pattern]
  ↓
[Filter: id > last_seen_id]
  ↓
[Router] ─┬─ 학사   → Sheets Add a Row → Telegram 🎓
          ├─ 공모전 → Sheets Add a Row → Telegram 🏆
          ├─ 기타   → Sheets Add a Row
          └─ i=1    → Sheets Update a Cell (last_seen_id 갱신)
```

### 4.2 n8n 워크플로우

```
[Schedule 09:00]
  ↓
[SsoCookie: HTTP Request]   Follow Redirects: Off / Include Response Headers: On / Never Error: On
  ↓
[GetState: Google Sheets]   _state_n8n 전체 3행
  ↓
[SrcConfig: Edit Fields]    JS 객체 룩업으로 board_url / base_url 계산
  ↓
[FetchBoard: HTTP Request]  Cookie 헤더 수동 매핑
  ↓
[HTML: Extract]             hrefs / titles / dates 배열
  ↓
[Split Out] → [Filter: 고정공지 제거] → [Edit Fields: 정규화]
  ↓
[NewOnly: Filter]           id > last_seen_id (소스별 대조)
  ↓
[Category: Switch] ─┬─ 학사     → Sheets Append → Telegram 🎓
                    ├─ 공모전   → Sheets Append → Telegram 🏆
                    └─ Fallback → Sheets Append
  ↓
[Summarize: source별 max(id)] → [Sheets Update Row]
```

### 4.3 파싱 방식 — 각 도구의 관용적 방식 채택

두 도구의 노드 개수와 파싱 수단은 다르지만 **입출력과 논리 흐름은 동일**하도록 설계하였습니다.

| | Make | n8n |
|---|---|---|
| 파싱 수단 | Text Parser (정규식) | HTML Extract (CSS 셀렉터) |
| 고정공지 제거 | 정규식 패턴에 의한 부작용 | Filter 노드 (명시적) |

**Make — Text Parser (Match Pattern), Global match: Yes**

```regex
<span class="title"><a href="read\.jsp\?id=(?<id>\d+)&code=[^"]+" title="(?<title>[^"]*)"(?:(?!<tr)[\s\S])*?<td class="date"[^>]*>(?<date>[\d.]+)</td>
```

`(?:(?!<tr)[\s\S])*?` 부분이 핵심입니다. 단순 `[\s\S]*?`를 사용하면 `<tr>` 경계를 넘어 다른 게시글 영역을 삼켜, **제목과 날짜가 서로 다른 게시글끼리 결합되는 오염**이 발생합니다. 실제 구현 중 겪은 문제입니다.

**n8n — HTML Extract (전체 문서에서 배열 추출)**

| Key | CSS Selector | Return Value | Attribute | Return Array |
|---|---|---|---|---|
| `hrefs` | `table td span.title a` | Attribute | `href` | On |
| `titles` | `table td span.title a` | Attribute | `title` | On |
| `dates` | `table td.date` | **HTML** | — | On |

`<tr>` 조각을 Split Out 후 개별 파싱하는 방식은 실패합니다. HTML 파서가 테이블 밖의 `<tr>`/`<td>`를 고아 태그로 보고 버리기 때문에 `td.date`가 매칭되지 않습니다. 실제로 `span.title a`는 살아남지만 `td.date`는 전부 빈 값이 되었습니다. 따라서 **전체 문서에서 배열로 추출**해야 합니다.

### 4.4 실행 결과 — 양쪽 완전 일치

| | Make | n8n |
|---|---|---|
| Split Out | — | 39건 (17+12+10) |
| 고정공지 제거 후 | 30건 | 30건 |
| 학사 | 16 (www 6 + cs 10) | 16 |
| 공모전 | 10 (swhit 10) | 10 |
| 기타 | 4 (www 4) | 4 |
| 텔레그램 발송 | 26건 | 26건 |
| 2회차 | 0건 | 0건 |

**파싱 경로가 다름에도 출력이 정확히 일치**하므로, "동일한 워크플로우 구조로 구현한다"는 요구를 데이터로 증명할 수 있습니다. 또한 두 도구가 서로의 결과를 검증하는 대조군 역할을 하였습니다.

### 5.2 항목별 상세

#### ① SSO 리다이렉트 처리 — 양쪽 동일 실패

`www.wsu.ac.kr/board/`는 익명 접근 시 SSO로 리다이렉트되며, 리다이렉트 체인이 4 hop이고 **중간에 세션 쿠키가 교체**됩니다.

```
[1] www/board          → 302  Set-Cookie: JSESSIONID=A     ← 쿠키 A 발급
      ↓ sso.wsu.ac.kr/svc/tk/Auth.eps?...
[2] sso Auth.eps       → 302  (Set-Cookie 없음)
      ↓ www/exsignon/sso/sso_identify.jsp?...failureCause=unauthorized&status=failure
[3] www/sso_identify   → 302  Set-Cookie: JSESSIONID=B     ← 쿠키 교체 (세션 고정 방어)
      ↓ www/board/index.jsp?code=community0101
[4] www/board          → 200
```

**Make와 n8n 모두 `Maximum number of redirects exceeded`로 실패**하였습니다. 두 도구 모두 Node.js 기반이며, 내부 HTTP 클라이언트가 리다이렉트 추적 중 `Set-Cookie` 재발급을 처리하지 않기 때문입니다.

대조군으로 curl(`-c/-b`)과 PowerShell(`-SessionVariable`)에서 동일 요청을 시도한 결과 **모두 성공**하였습니다. 따라서 이는 특정 도구의 결함이 아니라 **Node 계열 HTTP 클라이언트의 공통 제약**이라고 결론지었습니다.

**우회법(2 hop):** 1·2단계를 건너뛰고 `sso_identify.jsp`를 직접 호출하면 쿠키를 받을 수 있고, 그 쿠키로 게시판에 접근하면 200이 반환됩니다.

```
1단계 (Follow redirects OFF, 302에서 멈춰 Set-Cookie를 읽음):
https://www.wsu.ac.kr/exsignon/sso/sso_identify.jsp?RelayState=%2Fboard%2Findex.jsp%3Fcode%3Dcommunity0101&failureCause=unauthorized&status=failure

2단계: 위에서 받은 JSESSIONID를 Cookie 헤더에 실어 게시판 요청
```

양쪽 모두 동일한 우회를 적용하여 해결하였습니다. **도구 선택이 문제 해결에 아무런 영향을 주지 않은 사례**입니다.

#### ② 쿠키 공유 — Make 우위

| | Make | n8n |
|---|---|---|
| 방식 | `Share cookies with other HTTP modules` 옵션 On | 해당 옵션 없음 |
| 결과 | 이후 HTTP 모듈이 쿠키 자동 승계 | Cookie 헤더 수동 매핑 필수 |

n8n에서는 다음 표현식을 직접 작성해야 했습니다.

```javascript
{{ $('SsoCookie').first().json.headers['set-cookie'][0].split(';')[0] }}
```

Make는 체크박스 하나로 끝나는 작업이 n8n에서는 응답 헤더 구조를 이해하고 파싱하는 작업이 되었습니다. 다만 **n8n 방식은 어떤 쿠키를 어디에 싣는지 명시적**이므로, 의도치 않은 쿠키 전파가 없다는 장점도 있습니다.

#### ③ 분기 모델 — n8n 우위

**Make Router는 조건을 만족하는 모든 경로로 번들을 전송합니다.** if/else가 아닙니다. 따라서 배타적 분기를 구현하려면 부정 조건을 명시해야 했습니다.

| 경로 | Make 조건 | n8n 조건 |
|---|---|---|
| 학사 | 학사 키워드 매칭 | 학사 키워드 매칭 |
| 공모전 | 공모전 키워드 **AND NOT 학사 키워드** | 공모전 키워드 |
| 기타 | NOT 학사 **AND** NOT 공모전 | (Fallback Output) |
| **조건 총계** | **5개** | **2개** |

**n8n Switch는 기본이 first-match-only**이므로 규칙 순서만으로 배타성이 보장됩니다.

또한 Make의 **Fallback route는 다른 모든 경로가 실패했을 때만 실행**됩니다. 상태 갱신용 경로 4(`i=1`)를 추가하자 Fallback으로 지정한 기타 경로가 실행되지 않아 **매일 최신 공지 1건이 조용히 유실되는 버그**가 발생하였습니다. Fallback 지정을 해제하고 명시 조건으로 전환하여 해결하였습니다.

#### ④ 서드파티 인증 설정 — Make 우위

| | Make | n8n (자가호스팅) |
|---|---|---|
| 절차 | 클릭 2회 | GCP 프로젝트 생성 → Sheets+Drive API 활성화 → OAuth 동의 화면 구성 → 테스트 사용자 등록 → 클라이언트 발급 → 리디렉션 URI 등록 (6단계) |
| 소요 | **약 1분** | **약 20분** |
| 오류 | 없음 | `403: access_denied` 1회 (테스트 사용자 미등록) |

Make는 자체 OAuth 클라이언트를 보유한 검증된 앱이므로 사용자가 Google Cloud를 다룰 일이 없습니다. n8n 자가호스팅은 **사용자가 곧 앱 개발자**가 되므로 Google Cloud 콘솔 작업이 불가피합니다.

이는 자가호스팅의 구조적 특성이며, n8n의 결함이 아닙니다. 다만 **진입 장벽으로서 실재**합니다.

#### ⑤ 과금 모델 — n8n 우위

| | Make | n8n (자가호스팅) |
|---|---|---|
| 무료 범위 | 월 1,000 크레딧, 활성 시나리오 2개 | 무제한 |
| 과금 단위 | 모듈 실행 1회 = 1 크레딧 | — |
| 초기 적재 실측 | **73 크레딧** | 0 |
| 개발 중 누적 소모 | **155 크레딧** (잔여 845) | 0 |
| 정상 운영 | 11/일 → 약 330/월 | 0 |

주목할 점은 **번들 수만큼 후속 모듈에 크레딧이 전파**된다는 것입니다. 30건이 통과하면 이후 모듈이 30회 실행되므로, 모듈의 배치 위치가 곧 비용입니다.

또한 **개발 중 시행착오 실행도 과금 대상**입니다. 초기 적재 73 크레딧에 비해 누적 155가 소모된 것은 개발 과정의 반복 실행 때문입니다. n8n 자가호스팅에는 존재하지 않는 비용입니다.

정상 운영 기준 월 330 크레딧이므로 무료 플랜 1,000 범위 내에서 운영 가능하나, **실행 주기를 15분으로 올리면 하루 1,056 크레딧**이 되어 즉시 초과합니다. Make의 실행 주기가 사실상 1일 1회로 강제된 이유입니다.

#### ⑥ 데이터 흐름 모델 — 용도별

| | Make | n8n |
|---|---|---|
| 구조 | 선형 파이프라인 | 노드 그래프 |
| 특징 | 모듈 위치가 실행 횟수·비용에 직결 | 병렬 분기 자유 |
| 제약 | — | 표현식 참조 시 해당 노드가 **실행 계보상 조상**이어야 함 |

n8n에서 병렬 분기에 배치한 노드를 참조하려 하자 `There is no connection back to the node` 오류가 발생하였습니다. 그래프 구조라 해서 임의의 노드를 참조할 수 있는 것은 아닙니다.

Make는 선형이라 이런 혼란이 없는 대신, **분기 후 데이터를 다시 합치기 어렵습니다.**

#### ⑦ 표현식 언어 — 용도별

| | Make | n8n |
|---|---|---|
| 언어 | 자체 함수 세트 | 순수 JavaScript |
| 문자열 분할 | `split(text; ";")` (세미콜론 구분) | `text.split(';')` |
| 배열 인덱스 | **1-based** | **0-based** |
| 날짜 포맷 | `formatDate(now; "YYYY-MM-DD HH:mm")` | `$now.format('yyyy-MM-dd HH:mm')` |

날짜 포맷 문자열의 대소문자 규칙마저 다릅니다(`YYYY` vs `yyyy`). Make는 프로그래밍 경험이 없어도 접근 가능하나 함수 세트를 새로 익혀야 하고, n8n은 JavaScript를 안다면 즉시 쓸 수 있으나 모른다면 진입 장벽이 됩니다.

#### ⑧ 상태 갱신 설계 — Make 우위 (재검증으로 결론이 뒤집힌 항목)

**이 항목은 초기 평가와 최종 결론이 정반대입니다.** 그 과정을 그대로 기록합니다.

**초기 평가 (오판):**

| | Make | n8n |
|---|---|---|
| 방식 | Router 경로 4에 `i=1` 조건 → 첫 번째 번들의 id로 갱신 | `Sort(id Desc)` → `Limit(1)` → Update Row |
| 초기 평가 | 게시판이 최신순 정렬이라는 **가정에 의존** | 실제 최댓값을 산출하므로 **가정 불필요** |
| 당시 결론 | 열등 | **우월** |

**재검증에서 발견한 사실:**

작업 환경 이전 후 n8n 워크플로우를 재실행하자, 이미 수집한 20건이 신규로 재통과하는 현상이 관측되었습니다. `_state_n8n` 시트를 확인한 결과는 다음과 같았습니다.

| source | 값 | 해석 |
|---|---|---|
| www | 268169 | 갱신됨 |
| cs | **267454** | **초기값 그대로 — 한 번도 갱신된 적 없음** |
| swhit | 268123 | 이번 실행으로 갱신됨 |

**원인:** `Limit: 1`은 전체 스트림에서 단 1건만 남깁니다. 세 사이트가 ID 공간을 공유하고 www가 가장 빠르게 증가하므로, 전역 최댓값은 항상 www의 ID였습니다. 결과적으로 **매 실행마다 전역 1등 소스 하나만 갱신**되고 나머지는 초기값에 머물렀습니다.

**Make가 문제를 피한 이유:** Iterator가 소스별로 순회하므로 `i=1` 조건이 **소스마다 발동**합니다. 파이프라인 구조가 그룹 경계를 강제한 덕분에 우연히 정상 동작하였습니다.

**초기 평가가 오판이었던 이유:** 당시 검증이 www 단일 소스 기준이었습니다. www는 전역 최댓값을 가지므로 항상 갱신되었고, "2회차 0건"도 www 기준으로는 사실이었습니다. **관측 범위가 좁아 문제가 드러나지 않았습니다.**

**수정:** `Sort → Limit` 2개 노드를 **Summarize 1개**로 교체하였습니다.

```
수정 전:  [NewOnly] → [Sort: id Desc] → [Limit: 1] → [Update Row]
                                          ↑ 전역 1건

수정 후:  [NewOnly] → [Summarize: source별 max(id)] → [Update Row]
                        ↑ 소스별 1건, 항상 3건 이하
```

수정 후 재실행하여 세 소스 모두 갱신되는 것을 확인하였습니다.

| source | 초기값 | 갱신 후 |
|---|---|---|
| www | 268005 | 268169 |
| cs | 267454 | **267988** |
| swhit | 266262 | 268123 |

**최종 결론:** Make의 선형 파이프라인은 Iterator가 그룹 경계를 강제하여 설계자가 그룹핑을 의식하지 않아도 정상 동작하였습니다. n8n의 노드 그래프는 그 경계를 없애 주는 대신 **그룹 단위 집계를 설계자가 명시적으로 책임**져야 합니다. **자유도가 높은 만큼 실수의 여지도 큽니다.**

이 사례는 비교 항목 6(데이터 흐름 모델)의 구체적 귀결이기도 합니다.

#### ⑨ 타임존 — Make 우위

| | Make | n8n (자가호스팅) |
|---|---|---|
| 기본값 | 계정 프로필의 `Asia/Seoul` 자동 반영 | **`America/New_York`** |
| 결과 | 정상 | 기록 시각이 **13시간 오차** |

n8n의 텔레그램 실패 알림에 `시각: 2026-07-17 08:12`로 기록되었으나 실제 수신 시각은 21:12였습니다. KST(UTC+9)와 EDT(UTC−4)의 차이인 13시간입니다. Google Sheets의 `collected_at` 컬럼도 동일하게 어긋나 있었습니다.

**이 오류는 조용했습니다.** 예외가 발생하지 않고 잘못된 값이 기록될 뿐이라, 텔레그램 알림 시각을 사람이 눈으로 대조하지 않았다면 계속 발견되지 않았을 것입니다.

조치로 워크플로우별 Settings에서 Timezone을 `Asia/Seoul`로 명시하였습니다.

**해석:** 클라우드 도구는 계정 컨텍스트(타임존·로케일)가 가입 시점에 잡히지만, 자가호스팅은 호스트 OS 설정과 무관하게 애플리케이션 기본값을 따르므로 **명시 설정이 필요**합니다. 항목 4(인증 설정)와 같은 계열의 문제이며, "자가호스팅의 자유도에는 설정 책임이 따른다"는 일관된 특성을 보여줍니다.

## 6. 각 도구의 장단점 정리

### Make

**장점**
- 서드파티 인증 설정이 압도적으로 간편 (1분)
- 쿠키 공유 등 자주 쓰는 기능이 옵션으로 제공
- 클라우드 실행이므로 호스트 PC 가동과 무관
- 계정 컨텍스트(타임존 등)가 자동 반영
- 선형 구조가 파이프라인 경계를 강제하여 그룹핑 실수를 예방

**단점**
- 크레딧 과금이 설계를 제약 (실행 주기, 모듈 수, 개발 중 시행착오까지 과금)
- Router가 다중 매칭이라 배타 분기에 부정 조건 필요
- Fallback route의 동작이 직관과 어긋나 조용한 유실을 유발
- 자체 표현식 함수 세트를 새로 학습해야 함
- 분기 후 데이터 재결합이 어려움

### n8n (자가호스팅)

**장점**
- 실행 횟수 무제한 — 설계가 비용에서 자유로움
- Switch가 first-match-only라 분기 조건이 간결
- 순수 JavaScript 표현식 — 개발 경험이 있으면 즉시 활용
- Sort·Limit·Summarize 등 보조 노드 추가에 비용 부담 없음
- 노드 그래프 구조로 병렬 분기가 자유로움

**단점**
- 초기 환경 구축 부담 (Node.js 설치, 실행 정책, OAuth 6단계, 약 20분)
- 호스트 PC가 꺼지면 워크플로우가 멈춤
- 타임존 등 기본값이 한국 환경과 맞지 않아 명시 설정 필요
- 그룹 단위 집계를 설계자가 책임져야 함 (상태 갱신 버그의 원인)
- 자격증명·환경을 환경 이전 시마다 재구축해야 함

## 7. 어떤 상황에 적합한가

| 상황 | 권장 | 근거 |
|---|---|---|
| 실행 빈도가 낮고(일 1회 등) 즉시 가동이 필요 | **Make** | 인증 1분, 크레딧 여유, 클라우드 상시 가동 |
| 고빈도 실행(분 단위) 또는 대량 아이템 처리 | **n8n** | Make는 크레딧 초과 (15분 주기 시 1,056/일) |
| 비개발자가 단독 운영 | **Make** | JavaScript 불필요, 환경 구축 불필요 |
| 복잡한 데이터 가공·집계가 필요 | **n8n** | JS 표현식, Code 노드, 보조 노드 무료 |
| 24시간 안정 가동이 필수 | **Make** | 자가호스팅은 호스트 PC 가동이 전제 |
| 외부 서비스에 데이터를 두기 어려움 | **n8n** | 자가호스팅으로 데이터 통제 |
| 학습·실험 목적 | **n8n** | 시행착오에 비용이 들지 않음 |

**본 과제의 워크플로우(일 1회, 30건 처리)에 한정하면 Make가 더 적합합니다.** 크레딧 여유가 충분하고, 클라우드 상시 가동이라 노트북 전원과 무관하며, 파이프라인 구조가 상태 갱신 실수를 막아 주었기 때문입니다.

다만 프로젝트 2와 같이 **15분 주기 + AI 호출**로 확장하면 Make로는 무료 범위 내 구현이 불가능합니다. 따라서 프로젝트 2는 n8n을 선택하였습니다.

## 8. 설계 한계 (명시)

우연히 동작하는 것을 의도된 설계로 포장하지 않기 위해, 알려진 한계를 기록합니다.

| # | 한계 | 영향 |
|---|---|---|
| 1 | **Make의 고정공지 배제는 정규식의 우연한 부작용**입니다. 고정공지는 `<td class="date"><strong>` 구조라, `>` 뒤 숫자를 요구하는 정규식에 걸리지 않을 뿐입니다. | 학교가 CSS를 변경하면 파싱이 깨집니다. n8n은 Filter로 `<strong` 포함 여부를 명시 검사하므로 이 문제가 없습니다. |
| 2 | **Make의 `i=1` 상태 갱신은 게시판 최신순 정렬 가정에 의존**합니다. 30건 실측에서 모두 내림차순을 확인했으나 보장은 없습니다. Aggregator로 max(id)를 구하면 엄밀하나 모듈·크레딧이 증가하여 단순한 쪽을 택하였습니다. | 게시판 정렬이 바뀌면 상태 갱신이 오동작합니다. **과금 모델이 설계 품질을 낮춘 사례**입니다. |
| 3 | **키워드 분류는 한국어 2글자 한자어의 중의성으로 구조적 한계**가 있습니다. (상세는 프로젝트 2 문서 참조) | 규칙 개선으로 해결 불가능한 오분류가 존재합니다. |
| 4 | **n8n 자가호스팅은 호스트 PC 가동이 전제**입니다. | 노트북이 꺼지면 스케줄이 실행되지 않습니다. Make는 클라우드라 무관합니다. |
| 5 | 두 도구 모두 **SSO 2-hop 우회에 의존**합니다. | 학교가 SSO 흐름을 변경하면 양쪽 모두 깨집니다. |

## 9. 결론

두 도구는 **우열 관계가 아니라 트레이드오프 관계**입니다. 9개 비교 항목에서 Make가 5개, n8n이 2개 우위였고 2개는 용도에 따라 갈렸으나, 이 집계 자체는 큰 의미가 없습니다. 각 항목의 성격이 다르기 때문입니다.

구현을 통해 확인한 일관된 패턴은 다음과 같습니다.

**Make의 제약은 설계를 강제하고, n8n의 자유는 책임을 요구합니다.**

- Make는 크레딧이 실행 주기를 제한하고(항목 5), 그 제약이 설계 품질을 낮추기도 했습니다(한계 2). 반면 선형 구조가 그룹 경계를 강제하여 버그를 예방했습니다(항목 8).
- n8n은 비용 제약이 없어 Summarize·Sort 같은 노드를 자유롭게 쓸 수 있으나(항목 5), 그룹핑을 설계자가 책임져야 해 실제로 버그가 발생했습니다(항목 8). 타임존도 명시하지 않으면 조용히 틀립니다(항목 9).

가장 인상적이었던 것은 **SSO 리다이렉트 문제에서 두 도구가 동일하게 실패**한 점입니다(항목 1). 도구 선택으로 해결되지 않는 문제가 존재하며, 이때는 도구가 아니라 **프로토콜 수준의 이해**가 필요했습니다. curl과 PowerShell을 대조군으로 삼아 "도구 결함이 아닌 Node 계열 공통 제약"임을 확인한 과정이 그것입니다.

또한 **상태 갱신 항목의 결론이 재검증으로 뒤집힌 경험**(항목 8)은, 관측 범위가 좁으면 잘못된 비교 결론에 도달할 수 있음을 보여줍니다. www 단일 소스만 보았을 때 n8n이 우월해 보였으나, 세 소스 전체를 관측하자 정반대였습니다. 도구 비교에서 **어떤 조건으로 관측했는지가 결론만큼 중요**합니다.

## 부록 B. 그 밖의 실측 관찰

- n8n의 `Test step`은 시뮬레이션이 아니라 **실제 API를 호출**합니다. Sheets에 실제로 행이 기록됩니다. Make의 `Run once`도 동일합니다.
- n8n Edit Fields에서 `Fixed` 모드로 `{{ }}`를 입력하면 **리터럴 문자열로 저장**됩니다. Expression 토글 전환이 필수입니다. 이 실수로 `source` 값이 `"{{ $('SrcConfig').item.json.source }}"` 문자열이 되어 하위 필터가 전부 실패한 적이 있습니다.
- n8n의 `Execute Once: On`은 실행 횟수뿐 아니라 **출력 아이템도 1개로 축소**합니다. 체인 중간에 켜면 이후 전체가 1개로 붕괴합니다.
- 텔레그램 노드의 `Parse mode`는 비워야 합니다. 실측 제목에 `[코디세이 AI 올인원 과정]`, `「전국민 AI 경진대회」`, `(원금상환,이자지원)` 같은 특수문자가 있어 Markdown/HTML 모드에서 400 오류가 발생합니다.
- Make의 `Return error if HTTP request fails: Yes`는 4xx/5xx만 오류로 처리하므로 302는 통과합니다.
- Google Sheets는 `formatDate()` 결과 문자열을 날짜로 인식하지만, 열 서식이 숫자이면 **내부 일련번호**(`46218.78611`)로 표시됩니다. 데이터는 정상이며 서식만 조정하면 됩니다.
