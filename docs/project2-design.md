# 프로젝트 2 — 자유 주제 자동화 설계 및 구현

## 1. 자동화할 반복 업무 정의

### 업무 내용

우송대학교 재학생은 학사 일정과 교외 활동 정보를 얻기 위해 **서로 다른 3개의 공지사항 게시판**을 각각 방문해야 합니다.

| 게시판 | 성격 | 놓쳤을 때의 결과 |
|---|---|---|
| www.wsu.ac.kr (대학 공지) | 학사·행정 | 수강신청·휴학·장학금 신청 기한 경과 |
| cs.wsu.ac.kr (학부 공지) | 학사 행정 | 전공 관련 학사 안내 누락 |
| swhit.wsu.ac.kr (SW중심대학) | 공모전·교육 프로그램 | 선착순 모집 마감 |

### 반복성과 비용

| 항목 | 실측/추정 |
|---|---|
| 확인 주기 | 하루 1회 이상 |
| 1회 소요 | 3개 사이트 접속 + 신규 여부 판단 = 약 3~5분 |
| 연간 누적 | 약 18~30시간 |
| 실질 문제 | **확인을 걸러도 즉시 불이익이 없어 습관화되지 않음** |

이 업무의 핵심 문제는 시간이 아니라 **누락의 비대칭성**입니다. 매일 확인해도 대부분의 날은 볼 게 없지만, 하루를 거른 날에 선착순 모집 공고가 올라오면 기회를 잃습니다. 사람이 하기에는 보상이 희박하고 실패 비용만 큰, **자동화에 적합한 전형적 업무**입니다.

### 자동화 후

```
[변경 전] 사용자가 3개 사이트를 순회하며 신규 여부를 판단
[변경 후] 15분마다 시스템이 감시 → 신규 공지만 분류하여 텔레그램으로 push
```

사용자의 작업이 **능동적 확인(pull)에서 수동적 수신(push)으로 전환**됩니다.

## 2. 도구 선정 및 선정 이유

### 선정: n8n (로컬 자가호스팅)

프로젝트 1에서 Make와 n8n을 동일 조건으로 비교한 결과를 근거로 선정하였습니다.

| 선정 기준 | Make | n8n | 판정 |
|---|---|---|---|
| **15분 주기 실행** | 11 크레딧 × 96회 = **1,056/일** → 무료 1,000/월을 하루에 초과 | 무제한 | **n8n** |
| **AI 호출 Action 추가** | 모듈 추가 = 크레딧 추가. 30건 처리 시 30 크레딧 추가 | 비용 없음 | **n8n** |
| **응답 파싱·재가공 노드** | Code 모듈마다 번들 수만큼 과금 | 비용 없음 | **n8n** |
| 개발 중 시행착오 | **과금 대상** (프로젝트 1에서 155 크레딧 소모, 그중 82가 시행착오) | 비용 없음 | **n8n** |
| 상시 가동 | 클라우드 (우위) | 호스트 PC 가동 전제 | Make |
| 인증 설정 | 1분 (우위) | 20분 | Make |

**결정적 근거는 과금 모델입니다.** 프로젝트 1에서 실측한 "모듈 실행 1회 = 1 크레딧, 번들 수만큼 후속 모듈에 전파" 규칙에 따르면, 15분 주기 + AI 노드 추가 구성은 Make 무료 플랜에서 **구조적으로 불가능**합니다. 과제 제약이 "무료 플랜으로 완수 가능한 조합을 우선 고려한다"이므로 n8n이 유일한 선택지였습니다.

부차적으로, 프로젝트 2는 **AI 응답 파싱과 결과 재분배**를 위해 Code 노드를 3개 사용합니다. n8n은 순수 JavaScript를 쓸 수 있어 이 작업이 자연스럽습니다.

**감수한 단점:** n8n 자가호스팅은 호스트 PC가 꺼지면 멈춥니다. 이는 설계 한계로 명시합니다(8절).

## 3. 워크플로우 설계

### 3.1 전체 흐름

```
[Schedule Trigger: 15분]
  ↓
[SsoCookie: HTTP Request]        SSO 2-hop 우회 (Follow Redirects: Off)
  ↓
[GetState: Google Sheets]        _state_p2 3행 조회
  ↓
[SrcConfig: Edit Fields]         소스별 board_url / base_url 계산
  ↓
[FetchBoard: HTTP Request]       Cookie 헤더 수동 매핑        [Retry On Fail: 3회 / 2초]
  ↓
[HTML: Extract]                  hrefs / titles / dates 배열
  ↓
[Split Out]                      39건
  ↓
[Filter]                         dates에 '<strong' 미포함 → 고정공지 제거 → 30건
  ↓
[Edit Fields]                    id / title / date / source / url 정규화
  ↓
[NewOnly: Filter]                id > last_seen_id (소스별 대조)
  ↓
[RuleCategory: Code]             ← 키워드 규칙으로 category_rule 계산
  ↓
[BuildPrompt: Code]              ← 30건을 번호 매긴 목록 1건으로 병합
  ↓
[Basic LLM Chain] ── [Google Gemini Chat Model]   [Retry On Fail: 3회 / 30초]
  ↓                                                 ← 보너스 1: AI 연동 Action
[MergeAI: Code]                  ← 번호로 원본에 category_ai 부여 → 30건 복원
  ↓
[Category: Switch] ─┬─ 학사     → Sheets Append → Telegram 🎓
                    ├─ 공모전   → Sheets Append → Telegram 🏆
                    └─ Fallback → Sheets Append
  ↓
[Summarize: source별 max(id)] → [Sheets Update Row]

────────────────────────────────────────────────
[별도 워크플로우: error-handler]                  ← 보너스 2: 실패 알림
[Error Trigger] → [Telegram ⚠️]
```

### 3.2 Trigger / Action / 조건 분기

| 요구사항 | 구현 |
|---|---|
| Trigger 1개 이상 | Schedule Trigger (15분 주기, 자동 실행) |
| Action 2개 이상 | Google Sheets Append, Telegram Send Message, Google Sheets Update Row, Gemini 호출 |
| 조건 분기 1개 이상 | Filter(고정공지) / Filter(NewOnly) / **Switch(3경로)** |
| 각 분기 경로 1회 이상 실행 | 학사 16 / 공모전 10 / 기타(Fallback) 4 — 첫 실행에서 전부 발동 |

### 3.3 프로젝트 1과의 차이

| | 프로젝트 1 | 프로젝트 2 |
|---|---|---|
| 목적 | 도구 비교 (통제 실험) | 실사용 파이프라인 |
| 주기 | 09:00 1회 | **15분** |
| 분류 | 키워드 규칙만 | **키워드 + AI 병렬 산출** |
| 시트 탭 | `notices_n8n` / `_state_n8n` | `notices_p2` / `_state_p2` |
| category 컬럼 | 1개 | **`category_rule` / `category_ai` 2개** |
| 오류 처리 | 없음 | **Error Workflow + Retry** |

## 4. 보너스 1 — AI 연동 Action

### 4.1 배경: 키워드 규칙의 구조적 한계

프로젝트 1의 키워드 분류 규칙은 **4차에 걸쳐 개선**하였습니다. 검사 순서는 학사 → 공모전 → 기타로 고정입니다.

| 차수 | 오분류 사례 | 조치 | 결과 |
|---|---|---|---|
| 1차 | `미래이음 장학금 장학생 추가모집` (장학 + 모집 충돌) | 검사 순서를 공모전 우선 → **학사 우선으로 반전** | 해결 |
| 2차 | `첨단산업 인재양성 부트캠프(반도체)사업 교수초빙 공고` | 공모전 키워드에서 `캠프` 제거 | 해결 |
| 3차 | `충남대학교 수학 안내(학부 및 대학원)` | **수정 불가** | **미해결** |
| 4차 | `교과목 시간표 공지 (기말고사)` | 학사에 `고사`, `시간표` 추가 | 해결 |

**3차 사례가 핵심입니다.** `수학`을 학사 키워드에 추가하면 다음이 모두 오탐됩니다.

| 오탐 대상 | 실제 의미 |
|---|---|
| 수학여행 | 여행 |
| 수학능력시험 | 입시 |
| 수학과 | 학과명 |

반대로 추가하지 않으면 `충남대학교 수학(修學) 안내`가 기타로 분류되어 학사 공지가 누락됩니다. **한국어 2글자 한자어의 중의성은 키워드 매칭으로 해결할 수 없습니다.** 문맥이 필요합니다.

**패치할 때마다 새 구멍이 생기고, 3차는 구조적으로 해결 불가능**하다는 것이 4회 반복 개선 끝에 얻은 결론이며, AI 분류 도입의 근거입니다.

### 4.2 설계 방침: 대체가 아닌 병렬 비교

당초 계획은 Switch를 AI 분류로 **대체**하는 것이었으나, **키워드 분류를 남긴 채 AI 분류를 병렬로 산출**하는 방식으로 변경하였습니다.

| | 대체안 | **병렬 비교안 (채택)** |
|---|---|---|
| 정확도 비교 | 프로젝트 1 결과와 대조 (도구·시점이 다름) | **동일 실행·동일 30건에서 직접 대조** |
| `수학` 케이스 검증 | 간접 | 같은 행에서 `기타` vs `학사` 즉시 대비 |
| 기록 | category 1개 | `category_rule` / `category_ai` 2개 |

Switch는 그대로 두고 라우팅에만 사용하며(`category_rule` 기준), AI 결과는 컬럼으로 기록합니다. 두 분류 결과가 **같은 행에 나란히** 있으므로 대조가 즉각적입니다.

### 4.3 무료 티어 제약과 배치 프롬프트 전환

**Gemini 무료 티어(gemini-3-flash-preview)는 분당 5회로 제한**됩니다. 아이템 1건당 AI 호출 1회 방식으로 구현하자 첫 실행에서 즉시 오류가 발생하였습니다.

```
[429 Too Many Requests] Quota exceeded for metric:
generativelanguage.googleapis.com/generate_content_free_tier_requests,
limit: 5, model: gemini-3-flash
Please retry in 22.18423376s
```

30건이면 6분이 소요되며, 아이템 단위 호출은 이 제약과 구조적으로 맞지 않습니다.

| 대안 | 첫 실행 소요 | 호출 수 | 판정 |
|---|---|---|---|
| A. Retry On Fail만 적용 | 불규칙, 실패 가능 | 30+ | 백오프가 22초를 못 채우면 계속 실패 |
| B. Loop Over Items + Wait 13초 | 약 6.5분 | 30 | 노드 2개 추가, 구조 복잡 |
| **C. 배치 프롬프트 (채택)** | **수 초** | **1** | 노드 추가 없이 해결 |

**C안 채택.** 30건을 한 프롬프트에 담아 JSON 배열로 회신받으면 호출이 1회로 끝나고, 신규 건이 수십 건 몰려도 안전합니다.

> **관찰:** 프로젝트 1에서 "Make의 크레딧 과금이 설계 품질을 낮춘 사례"(Aggregator 대신 정렬 가정에 의존)를 기록하였는데, 프로젝트 2에서도 **무료 티어 제약이 자연스러운 설계를 불가능하게 만들어 우회 설계를 강제**하였습니다. 도구는 달라도 **비용 모델이 설계를 규정한다**는 패턴은 동일합니다.

### 4.4 구현 — Code 노드 3개

**① RuleCategory — 키워드 분류를 값으로 계산**

프로젝트 1에서는 Switch가 라우팅만 하고 `category` 값은 각 분기의 Append에 하드코딩되어 있었습니다. 즉 아이템 자체에는 분류 결과가 없었습니다. 두 결과를 한 행에 담으려면 규칙 분류를 **필드로 계산**해야 합니다.

```javascript
const RULE_B = /수강|휴학|복학|전과|재입학|시험|고사|시간표|성적|졸업|장학|출석|학점|등록금/;
const RULE_A = /경진대회|공모전|해커톤|대회|모집|창업|교육생/;

for (const item of $input.all()) {
  const t = item.json.title ?? '';
  item.json.category_rule = RULE_B.test(t) ? '학사'
                          : RULE_A.test(t) ? '공모전'
                          : '기타';
}

return $input.all();
```

삼항 연산자의 평가 순서가 곧 검사 순서(학사 → 공모전 → 기타)입니다. 프로젝트 1 Switch와 동일한 결과(학사 16 / 공모전 10 / 기타 4)를 내는 것을 확인하였습니다.

**② BuildPrompt — 30건을 1건으로 병합**

```javascript
const items = $input.all();

const list = items
  .map((it, i) => `${i + 1}. ${it.json.title ?? ''}`)
  .join('\n');

return [{ json: { batch_prompt: list, count: items.length } }];
```

출력이 1건이 되어 AI 호출도 1회로 수렴합니다.

**③ 프롬프트**

```
다음은 대학교 공지사항 제목 목록입니다. 각 항목을 아래 세 가지 중 하나로 분류하십시오.

- 학사: 수강, 휴학, 복학, 전과, 시험, 성적, 졸업, 장학, 등록금 등 재학생의 학사 행정에 관한 것
- 공모전: 경진대회, 공모전, 해커톤, 교육 프로그램, 창업 지원 등 참가 신청이 필요한 대외 활동
- 기타: 위 둘 중 어디에도 해당하지 않는 것

규칙:
- 입력 항목 수와 출력 배열의 길이가 반드시 같아야 합니다.
- 순수 JSON 배열만 출력하십시오. 설명, 마크다운 코드펜스(```)를 붙이지 마십시오.
- 출력 형식: [{"n":1,"category":"학사"},{"n":2,"category":"공모전"}]

목록:
{{ $json.batch_prompt }}
```

키워드 규칙과 **동일한 기준**을 자연어로 제시하였습니다. AI에게 유리한 추가 정보를 주지 않아야 공정한 비교가 됩니다.

**④ MergeAI — 원본 복원 및 결과 부여**

```javascript
const raw = $input.first().json.text ?? '';

// 코드펜스나 앞뒤 군더더기가 붙어도 배열만 추출
const m = raw.match(/\[[\s\S]*\]/);
if (!m) throw new Error('AI 응답에서 JSON 배열을 찾지 못했습니다: ' + raw.slice(0, 200));

const parsed = JSON.parse(m[0]);
const map = new Map(parsed.map(o => [Number(o.n), o.category]));

const origin = $('RuleCategory').all();
const VALID = ['학사', '공모전', '기타'];

return origin.map((it, i) => {
  const ai = map.get(i + 1);
  return {
    json: {
      ...it.json,
      category_ai: VALID.includes(ai) ? ai : '분류실패',
    },
  };
});
```

방어 설계 3가지를 포함하였습니다.

| 방어 | 목적 |
|---|---|
| `raw.match(/\[[\s\S]*\]/)` | AI가 코드펜스나 설명을 덧붙여도 배열만 추출 |
| `throw new Error(...)` | 배열을 못 찾으면 **명시적 실패** → Error Workflow 발동 |
| `VALID.includes(ai) ? ai : '분류실패'` | 예상 밖 값을 조용히 통과시키지 않고 눈에 띄게 표시 |

`$('RuleCategory').all()`로 원본 30건을 되찾아 붙이므로, LLM Chain이 출력을 `{ text: ... }`로 교체해도 필드가 유실되지 않습니다.

### 4.5 결과 — 규칙 29/30 → AI 30/30

**독립 실행 2회에서 동일한 결과가 재현되었습니다.**

| 항목 | 값 |
|---|---|
| 대상 | 30건 |
| 두 분류 일치 | **29건** |
| 불일치 | **1건** |
| 불일치 건 | `2026학년도 제2학기 충남대학교 수학 안내(학부 및 대학원)` |
| `category_rule` | 기타 (오답) |
| `category_ai` | **학사 (정답)** |

**정확히 3차 개선에서 "수정 불가"로 판정했던 그 사례입니다.**

AI는 키워드가 아니라 **문맥**을 보았습니다. `2026학년도`, `제2학기`, `학부 및 대학원`이라는 주변 단서로부터 이것이 타 대학 수학(修學) 안내, 즉 학사 행정임을 판단하였습니다. `수학`이라는 두 글자만으로는 결정 불가능한 중의성이 문맥에서 해소되었습니다.

**더 중요한 것은 퇴행이 없었다는 점입니다.**

| | 키워드 규칙 | AI 분류 |
|---|---|---|
| 개선 방식 | 패치할수록 새 구멍 (4차까지 반복) | 규칙 추가 없음 |
| 기존 정답 29건 | 정답 | **정답 유지** |
| `수학` 케이스 | **구조적 해결 불가** | 해결 |
| 정확도 | 29/30 (96.7%) | **30/30 (100%)** |

수치상으로는 3.3%p 개선에 불과하지만, **규칙 개선이 매번 겪던 트레이드오프가 없었다**는 점이 본질입니다. "애매한 하나를 맞추는 대신 멀쩡하던 것들이 틀리는" 현상이 발생하지 않았습니다.

## 5. 보너스 2 — 실패 알림 및 재시도 전략

### 5.1 실패 알림 — Error Workflow

n8n은 오류 알림을 **별도 워크플로우**로 구현합니다.

```
[error-handler]
[Error Trigger] → [Telegram Send Message]
```

**메시지 템플릿**

```
⚠️ 워크플로우 실행 실패

워크플로우: {{ $json.workflow.name }}
실패 노드: {{ $json.execution.lastNodeExecuted }}
오류: {{ $json.execution.error.message }}
시각: {{ $now.format('yyyy-MM-dd HH:mm') }}
```

**`Parse mode`는 반드시 비웁니다.** 오류 메시지에는 대괄호·따옴표·백틱이 섞여 있어 Markdown/HTML 모드에서 400 오류가 발생합니다. **알림 자체가 실패하면 의미가 없으므로** 이 설정이 특히 중요합니다.

각 워크플로우의 Settings → Error Workflow에 `error-handler`를 지정하였습니다.

### 5.2 재시도 — Retry On Fail

| 노드 | Retry On Fail | Max Tries | Wait Between Tries | 근거 |
|---|---|---|---|---|
| **Basic LLM Chain** | On | 3 | **30,000 ms** | 429 오류가 `Please retry in 22.18s`를 명시 → 그보다 길어야 함 |
| **FetchBoard** | On | 3 | 2,000 ms | 네트워크 순단은 짧은 대기로 충분 |

**대기 시간 30초는 실측 오류 메시지에서 역산한 값입니다.** n8n 기본값 1초로는 세 번 모두 실패합니다.

재시도가 실제로 동작한 증거는 실행 이력에 남았습니다. 정상 실행이 1~2초인 반면, 재시도가 발생한 실행은 **37.244s / 46.871s / 58.383s**로 기록되었습니다.

### 5.3 실측 검증 — 의도적 실패 유도

Gemini API 키를 잘못된 값으로 변경한 뒤 자동 실행을 기다려 전 과정을 검증하였습니다.

| 검증 항목 | 결과 |
|---|---|
| 재시도 발동 | 실행 시간이 1초대 → 30초 이상으로 증가 |
| 워크플로우 실패 기록 | Executions에 `Error` (Exec ID 17) |
| Error Workflow 발동 | `error-handler` `Success` (Exec ID 18), **1초 후** |
| 텔레그램 알림 수신 | ⚠️ 메시지 도착 |

### 5.4 실측으로 확인한 n8n의 동작 특성

구현 과정에서 문서화되지 않은 동작을 두 가지 확인하였습니다.

**① Error Workflow는 자동 실행에서만 발동합니다.**

에디터에서 `Test workflow`로 수동 실행하여 실패시켰을 때는 알림이 오지 않았습니다. 처음에는 설정 오류를 의심했으나, 확인 결과 다음과 같았습니다.

| 관측 | 해석 |
|---|---|
| Executions에 `Error`로 기록됨 | 실행은 시작되었고 실패로 끝남 |
| LLM Chain 노드에 실패 표시 | 앞 단계는 정상, AI 호출에서 401 |
| 텔레그램 미도착 | **수동 실행이라 Error Workflow 미호출** |

n8n은 개발 중 시행착오가 정상이라고 보아 수동 실행에는 알림을 보내지 않습니다. **설정 오류가 아니라 의도된 동작**입니다. 따라서 검증을 위해 Schedule Trigger 전환을 선행해야 했습니다.

실행 이력 캡처에서 이 사실이 그대로 드러납니다. 실패한 실행 중 **Exec 17에만 수동 실행 표식이 없고**, `error-handler`(Exec 18)는 그 직후 1초 만에 발동하였습니다. 다른 실패 실행(수동)에는 알림이 없습니다.

**② Error Workflow로 지정하려면 대상이 Publish 상태여야 합니다.**

`error-handler`를 저장만 한 상태에서는 Settings의 Error Workflow 목록에서 선택이 차단(금지 표시)되었습니다. Publish 후 선택 가능해졌습니다.

## 7. 구현 중 겪은 문제와 해결

| # | 문제 | 원인 | 해결 |
|---|---|---|---|
| 1 | `429 Too Many Requests` | Gemini 무료 티어 5 RPM, 아이템당 1호출 = 30회 | **배치 프롬프트로 전환** (호출 1회) |
| 2 | `Column names were updated after the node's setup` | 복제본이 `notices_n8n`을 가리킨 채 탭만 `notices_p2`로 변경 → 헤더 불일치 | Sheets 노드 5개의 탭 지정 및 컬럼 매핑 갱신 |
| 3 | `302 - The URL has moved` | `Test step`이 상위 노드의 **캐시된 출력**을 재사용 → 만료된 JSESSIONID 사용 | 전체 실행으로 쿠키 재취득 |
| 4 | `The execution took a different path` | GetState가 `_state_n8n`(갱신 완료)을 읽어 NewOnly에서 0건 → 하위 노드 미실행 | 탭을 `_state_p2`로 교체 |
| 5 | Error Workflow 선택 차단 | `error-handler`가 Publish되지 않음 | Publish 후 선택 가능 |
| 6 | 알림 시각이 실제와 13시간 차이 | n8n 기본 타임존이 `America/New_York` | 워크플로우 Timezone을 `Asia/Seoul`로 명시 |

**3번과 4번은 같은 뿌리입니다.** n8n의 `Test step`은 상위 노드를 재실행하지 않고 이전 출력을 재사용하므로, 개발 중에는 편리하지만 세션·상태에 의존하는 워크플로우에서는 오해를 유발합니다.

**6번은 조용한 오류였습니다.** 예외가 발생하지 않고 잘못된 값이 기록될 뿐이라, 텔레그램 알림 시각(08:12)과 실제 수신 시각(21:12)을 사람이 눈으로 대조하지 않았다면 발견되지 않았을 것입니다.

## 8. 설계 한계

우연히 동작하는 것을 의도된 설계로 포장하지 않기 위해 기록합니다.

| # | 한계 | 상세 |
|---|---|---|
| 1 | **AI 분류는 비결정적입니다.** | 30/30은 "이 표본에서 2회 재현된 관측"이지 정확도 100% 보장이 아닙니다. 표본은 30건, 단일 시점입니다. |
| 2 | **배치 프롬프트는 항목 수 불일치 위험이 있습니다.** | AI가 30개를 넣었는데 29개를 반환하면 `map.get(i+1)`이 `undefined`가 되어 해당 건이 `분류실패`로 표시됩니다. 조용한 오분류는 막지만 그 건은 잃습니다. 프롬프트에 길이 일치를 명시했으나 강제할 수단은 없습니다. |
| 3 | **AI 응답 파싱은 정규식 추출에 의존합니다.** | `/\[[\s\S]*\]/`는 응답에 대괄호가 하나뿐이라고 가정합니다. AI가 설명 중에 다른 배열을 언급하면 오작동할 수 있습니다. |
| 4 | **무료 티어 5 RPM은 확장 여지가 없습니다.** | 배치 전환으로 호출은 1회가 되었으나, 신규 공지가 수백 건 몰리면 프롬프트가 길어져 다른 제약(토큰 한도)에 걸립니다. |
| 5 | **n8n 자가호스팅은 호스트 PC 가동이 전제입니다.** | 노트북이 꺼지면 15분 주기가 멈춥니다. Make는 클라우드라 무관하나, 과금 때문에 선택할 수 없었습니다. |
| 6 | **키워드 규칙(`category_rule`)은 여전히 라우팅 기준입니다.** | AI 결과는 기록만 되고 텔레그램 발송 경로를 결정하지 않습니다. AI가 비결정적이므로 라우팅을 맡기지 않은 보수적 설계이나, 결과적으로 `수학` 케이스는 여전히 기타로 라우팅됩니다. |
| 7 | **SSO 2-hop 우회에 의존합니다.** | 학교가 SSO 흐름을 변경하면 www 소스 수집이 중단됩니다. |
| 8 | **파싱은 학교 HTML 구조에 의존합니다.** | `td.date`의 `<strong>` 유무로 고정공지를 구분하므로, CSS가 바뀌면 필터가 깨집니다. |

**6번은 의도적 선택이나 절충입니다.** AI 라우팅으로 전환하면 `수학` 케이스가 학사로 발송되지만, AI 응답이 실패한 날 전체 분류가 마비됩니다. 현재 설계는 **AI가 실패해도 규칙 기반으로 서비스가 계속되는** 구조이며, AI 결과는 관찰·비교용입니다. 향후 AI 안정성이 충분히 검증되면 라우팅 기준을 전환하고 규칙을 폴백으로 남기는 것이 자연스러운 발전 방향입니다.

## 9. 결론

| 목표 | 달성 |
|---|---|
| 반복 업무 자동화 | 3개 게시판 15분 주기 감시 → 텔레그램 push 전환 |
| 도구 선정 근거 | 프로젝트 1의 크레딧 실측(11/실행)에 근거하여 n8n 선정 |
| 자동 실행 | Schedule Trigger 15분, 실행 이력으로 확인 |
| 조건 분기 3경로 발동 | 학사 16 / 공모전 10 / 기타 4 |
| 보너스 1 (AI 연동) | 키워드 29/30 → AI 30/30, 2회 재현 |
| 보너스 2 (실패 알림·재시도) | Error Workflow + Retry On Fail, 의도적 실패로 검증 |

프로젝트 2에서 얻은 가장 큰 소득은 **"규칙 기반의 한계를 실측으로 증명한 뒤 AI로 넘어갔다"는 순서**입니다. 처음부터 AI를 썼다면 `충남대학교 수학 안내` 한 건을 맞춘 것이 우연인지 실력인지 알 수 없었을 것입니다. 4차에 걸친 규칙 개선과 3차의 "수정 불가" 판정이 있었기에, AI의 문맥 이해가 무엇을 해결했는지 정확히 특정할 수 있었습니다.

동시에 **AI가 만능이 아니라는 점도 설계에 반영**하였습니다. 비결정성 때문에 라우팅은 여전히 규칙이 담당하고, AI 결과는 기록·비교용입니다(한계 6). AI 응답 파싱에는 방어 코드를 두었고, 실패 시 Error Workflow가 알립니다.

프로젝트 1에서 "Make의 크레딧이 설계 품질을 낮춘 사례"를 관찰했는데, 프로젝트 2에서도 **Gemini 무료 티어 5 RPM이 아이템 단위 호출이라는 자연스러운 설계를 불가능하게 만들어 배치 우회를 강제**했습니다. 도구와 벤더가 달라도 **비용 모델이 설계를 규정한다**는 패턴은 반복되었습니다. 무료 도구로 자동화를 구축한다는 것은, 기능의 제약보다 **비용 구조에 맞춰 설계를 재발명하는 일**에 가까웠습니다.
